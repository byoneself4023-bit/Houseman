# Houseman 재구축

## PWA 방향성 (앱처럼 쓰기)

Flutter나 React Native 같은 네이티브 앱은 이 프로젝트에서 불필요하다. 사용자가 8명 내부 직원이고, 앱스토어에 배포할 서비스가 아니기 때문이다.

지금 만들고 있는 건 **반응형 웹앱**이다. 모바일 브라우저에서 그대로 쓸 수 있고, 바텀탭이 있는 이유가 직원들이 현장에서 스마트폰으로 접속할 때를 위한 것이다.

"앱처럼" 쓰고 싶으면 Phase 6(DevOps)에서 **PWA(Progressive Web App)** 설정을 추가하면 된다. 필요한 건 딱 2개:

**manifest.json** — 앱 이름, 아이콘, 테마 색상, 시작 URL을 정의하는 JSON 파일. 이게 있으면 모바일 브라우저에서 "홈 화면에 추가" 했을 때 앱 아이콘이 생기고, 실행하면 주소창 없이 전체 화면으로 뜬다.

**Service Worker** — 백그라운드에서 돌아가는 스크립트. 네트워크 요청을 가로채서 캐시에서 응답할 수 있어서, 오프라인이거나 네트워크가 불안정해도 앱이 뜬다. 직원들이 건물 현장(지하, 엘리베이터 등)에서 접속할 때 유용하다.

Phase 6에서 `vite-plugin-pwa` 하나 설치하면 둘 다 자동 생성된다. 코드 2~3줄 추가 수준. 8명 직원한테 "이 링크 열어서 홈 화면에 추가하세요" 한마디면 끝이다.

Flutter/React Native가 필요한 시점은 "카메라로 계량기 사진 찍어서 OCR로 자동 입력", "푸시 알림", "오프라인 모드" 같은 네이티브 기능이 필수일 때인데, 지금 Houseman 요구사항에는 없다.

## FE 툴링 기반

Phase 0에서 한 걸 비유하면 — **집은 그대로 두고 배관, 전기, 기초 공사를 한 것이다.**

### 뭘 한 건가

대표님이 만든 프로토타입은 JavaScript로 되어있었다. JavaScript는 "자유로운" 언어라 오타를 써도, 잘못된 데이터를 넣어도 실행하기 전까지 모른다. 실행해봐야 "터진다".

Phase 0에서 **TypeScript**를 씌웠다. 이건 "실행 전에 실수를 잡아주는 안전망"이다.

예를 들어:

```
// JavaScript — 실행해봐야 터지는 거 알 수 있음
tenant.bulding  // ← "building" 오타인데 아무도 안 잡아줌. 런타임에 undefined

// TypeScript — 코드 쓰는 순간 빨간줄
tenant.bulding  // ← 빨간줄: "bulding"은 Tenant에 없습니다. "building" 아닌가요?
```

42개 건물, 500명 임차인, 정산 설정 같은 **모든 데이터 구조를 타입으로 정의**했다 (434줄). 이제 이 데이터를 다루는 코드에서 오타나 잘못된 접근을 하면 빌드 자체가 안 된다. 실행해보기 전에 잡아준다.

### 같이 설치한 도구들

**ESLint** — "이 코드 이상한데?" 알려주는 도구. 안 쓰는 변수, React 훅 잘못 쓴 것, 위험한 패턴 등을 자동 감지한다.

**Prettier** — 코드 스타일 자동 통일. 들여쓰기, 따옴표, 세미콜론 같은 걸 한 번에 맞춰준다. 여러 명이 코드를 만져도 스타일이 동일하게 유지된다.

**Vitest** — 자동 테스트. "초성 검색이 잘 되나?", "금액 포맷이 맞나?" 같은 걸 코드가 자동으로 확인해준다. 14개 테스트가 돌아가고, 코드를 바꿀 때마다 "기존 기능이 안 깨졌는지" 자동 확인한다.

### 뭘 안 건드렸나

대표님이 만든 25개 페이지(임차인 관리, 건물 상세, 캘린더 등)와 App.jsx는 **한 줄도 안 건드렸다.** 데이터 값도 안 바꿨다. "안전망을 까는 작업"이지 "기능을 바꾸는 작업"이 아니다.

## FE 아키텍처 스켈레톤

Phase 1을 비유하면 — **방 하나짜리 원룸을 거실/침실/주방/욕실로 나눈 것이다.** 집 크기는 같은데 구조가 생겼다.

### 뭘 한 건가

대표님이 만든 App.jsx는 780줄짜리 단일 파일이었다. 로그인, 사이드바, 상단 바, 모바일 탭, 페이지 전환, 데이터 관리가 전부 한 파일에 들어있었다. 이걸 6개 파일로 쪼갰다:

- **AuthGate.tsx** — 로그인 화면
- **Sidebar.tsx** — 왼쪽 사이드바 메뉴
- **TopBar.tsx** — 상단 바 (사용자 정보, 시간)
- **MobileNav.tsx** — 모바일 하단 탭
- **AppLayout.tsx** — 전체 레이아웃 조합
- **router.tsx** — 페이지 경로 정의

### 사용자가 체감하는 차이 4가지

**1. URL이 생겼다**

이전에는 어떤 페이지를 가든 주소창이 `localhost:5173`으로 고정이었다. 지금은 사이드바에서 메뉴를 클릭하면 `/calendar`, `/tenants`, `/buildings` 이런 식으로 주소가 바뀐다.

**2. 뒤로가기가 된다**

이전에는 브라우저 뒤로가기를 누르면 아예 다른 사이트로 나갔다. 지금은 임차인정보 → 건물·호실정보로 이동한 뒤 뒤로가기를 누르면 임차인정보로 돌아온다.

**3. URL 직접 입력이 된다**

이전에는 `localhost:5173` 외에 아무 주소도 먹히지 않았다. 지금은 주소창에 `localhost:5173/tenants`를 직접 치면 바로 임차인 페이지로 간다. 이 URL을 다른 사람한테 공유할 수도 있다.

**4. 새 탭에서 열기가 된다**

이전에는 불가능했다. 지금은 `localhost:5173/buildings`를 새 탭에 붙여넣으면 건물 페이지가 바로 뜬다.

### 뭘 안 건드렸나

25개 페이지의 내부 코드는 한 줄도 안 건드렸다. 데이터 흐름(localStorage → 화면 표시)도 그대로다. "방을 나눈 것"이지 "가구를 바꾼 것"이 아니다.

### 레이아웃 세련화 

pages를 건드리지 않고 레이아웃 3파일(로그인, 사이드바, 상단 바)만 수정해서 첫인상을 바꿨다.

**로그인 화면** — 기존 1단 카드에서 2단 레이아웃으로 바뀌었다. 좌측에 브랜드 영역(HOUSEMAN 로고 + "건물관리의 새로운 기준" 슬로건), 우측에 로그인 폼. 모바일에서는 자동으로 1단으로 전환된다.

**사이드바** — 현재 보고 있는 메뉴 옆에 파란색 세로 바가 표시된다. 이전에는 작은 파란 점이었는데 더 눈에 잘 띈다. 메뉴 간격이 약간 넓어져서 답답한 느낌이 줄었다.

**상단 바** — "박종호 대표" 텍스트 옆에 이니셜이 든 원형 아바타가 추가됐다. 우측에 있던 "🏗️ 관리" 표시는 삭제됐다 (사이드바 역할 탭으로 충분하므로). 알림 벨 아이콘 위에 마우스를 올리면 살짝 흔들린다.

## 백엔드 기본 인프라

Phase 3a를 비유하면 — **건물에 수도관과 전기선을 연결한 것이다.** 건물(프론트엔드)은 그대로인데, 뒤에서 물(데이터)이 흐를 수 있는 배관(서버)이 깔렸다.

### 뭘 한 건가

지금까지 모든 데이터는 브라우저의 localStorage에 저장됐다. 이건 "내 컴퓨터에만 있는 메모장"이다. 다른 직원이 볼 수 없고, 브라우저 캐시를 지우면 데이터가 사라진다.

Phase 3a에서 **Spring Boot 서버**를 만들었다. 이 서버가 **PostgreSQL 데이터베이스**에 데이터를 저장한다. 직원 9명, 건물 21개, 호실 200개를 데이터베이스에 넣었다. 값은 프로토타입과 정확히 동일하다.

### 서버가 하는 일

**로그인** — 전화번호와 비밀번호를 서버에 보내면, 서버가 확인하고 "출입증"(JWT 토큰)을 발급한다. 이 출입증이 있어야 데이터에 접근할 수 있다. 프로토타입에서는 비밀번호가 코드에 그대로 노출돼 있었는데, 이제 암호화(BCrypt)된 상태로 데이터베이스에 저장된다.

**건물/호실 조회** — 출입증을 제시하면 자기가 담당하는 건물 목록을 받을 수 있다. 총괄 직원은 전체 21개, 담당자는 자기 건물만 보인다.

### 아직 프론트에는 연결 안 됐다

서버는 만들어졌지만, 프론트엔드는 아직 localStorage를 보고 있다. Phase 5에서 스위치를 켜면(VITE_USE_API=true) 프론트가 서버에서 데이터를 가져오게 된다.

## 핵심 비즈니스

Phase 3b를 비유하면 — **건물 배관에 실제 물을 흘려보낸 것이다.** Phase 3a에서 배관(서버)을 깔았고, 이제 계약/거래/청구라는 실제 업무 데이터가 흐르기 시작했다.

### 뭘 한 건가

프로토타입에서 가장 중요한 3가지 데이터를 서버로 옮겼다.

**계약** — 임차인 488명의 계약 정보(이름, 전화번호, 보증금, 월세, 관리비, 입주일, 만료일, 연체 상태). 퇴실 기록 14건도 포함.

**거래** — 입금/지출 내역 7건. 어떤 건물, 어떤 호실에서 얼마가 들어오고 나갔는지.

**청구** — 이 부분이 가장 복잡하다. 42개 건물마다 정산 방식이 다르다(수수료율 5%, 월급형 150만원, 고정형 50만원 등). 호실마다 전기/가스/수도 설정이 다르다(계량기 번호, 단가, 검침 기간 등). 이 설정을 전부 데이터베이스에 넣었다.

### 정산 계산 로직 보존

대표님이 만든 billingMaster.ts의 계산 함수 5개를 Kotlin으로 그대로 옮겼다. 10원 단위 절사, 연체수수료(납기 5일 초과 시 5%), 수수료 계산, 일할 계산, 부가세 계산. 한전 누진 3구간 요금 계산도 포함. **같은 숫자를 넣으면 같은 결과가 나온다.**

### 청구서 생성이 서버에서 된다

이전에는 브라우저에서 계산했다. 이제는 서버에 "제이앤제이 2026년 3월 청구서 만들어줘"라고 요청하면, 서버가 해당 건물의 계약 정보 + 공과금 설정을 조합해서 호실별 청구서를 자동 생성한다. 생성(DRAFT) → 확정(CONFIRMED) → 발송(SENT) 단계로 관리.

## 파생 도메인 + 실시간 알림

Phase 3c를 비유하면 — **건물에 인터폰과 CCTV를 설치한 것이다.** 배관(서버)으로 물(데이터)이 흐르고 있는 상태에서, 이제 "누가 입주했다", "연체가 발생했다" 같은 알림이 실시간으로 울리게 됐다.

### 뭘 한 건가

**캘린더** — 입퇴실 일정과 직원 휴무를 관리한다. "3월 1일 아페이론 101호 퇴실", "3월 7일 유은혜 부장 휴무" 같은 이벤트를 등록/수정/삭제할 수 있다.

**공실 관리** — 비어있는 호실을 등록하고 상태(점검/청소중, 홍보중 등)를 관리한다. 보증금, 월세, 중개 수수료 정보도 포함.

**건물주 정산** — 이 부분이 가장 복잡하다. 건물마다 정산 방식이 다르다. 수수료형(5%), 월급형(150만원), 수금형 등. 서버에 "스타빌 2026년 3월 정산해줘"라고 요청하면, 해당 건물의 16개 호실 임대료에서 수수료를 계산하고, 이번 달 입퇴실 정산, 공과금 공제까지 반영한 최종 정산서를 만들어준다.

**실시간 알림(SSE)** — 직원이 로그인하면 서버와 연결이 유지된다. 다른 직원이 계약을 등록하면 "스타빌 105호 입주 예정: 홍길동" 알림이 즉시 뜬다. 청구서가 확정되면, 입금이 들어오면, 연체가 감지되면 — 모두 실시간 알림.

**자동 감시** — 매일 아침 9시에 서버가 연체 계약을 자동 감지해서 알림을 보낸다. 매월 1일에는 30일 이내 만료되는 계약을 알려준다. 직원이 따로 확인할 필요 없이 서버가 알아서 챙겨준다.

## 부가 기능

Phase 3d를 비유하면 — **건물에 우편함과 주차장 관리대를 놓은 것이다.** 핵심 배관(정산, 청구, 계약)은 이미 완성됐고, 부가 시설을 추가한 것이다.

### 뭘 한 건가

**출납장(Cashbook)** — 건물별 입금/출금을 기록하는 장부. 정산 후 건물주에게 송금할 때, 수리비를 지출할 때 등을 기록한다. "대기 → 완료 → 보류" 상태 관리가 있어서 송금 전/후를 추적할 수 있다.

**주차 관리** — 임차인별 차량 정보(차량번호, 차종)를 관리한다. 어떤 건물, 어떤 호실의 임차인이 어떤 차를 가지고 있는지 기록.

**엑셀 업로드는 스킵** — 대표님이 만든 엑셀 업로드 기능은 브라우저에서 직접 localStorage에 쓰는 방식이라 서버 API가 필요 없다. Phase 5에서 서버 연결할 때 같이 처리.

### Phase 3 전체가 끝났다

Phase 3a(기본 인프라) → 3b(계약/거래/청구) → 3c(캘린더/공실/정산/실시간 알림) → 3d(출납/주차)까지 완료. 서버에 53개 API가 만들어졌고, 98개 테스트가 전부 통과한다. 프론트엔드는 한 줄도 안 건드렸다.

## Backend API 마무리

Phase 4를 비유하면 — **건물 준공 검사를 한 것이다.** 배관(서버), 전기(인증), 인터폰(실시간 알림), 우편함(출납), 주차장(주차 관리)까지 다 설치한 뒤, "입주부터 퇴실까지 전체 흐름이 정상 작동하는가"를 테스트했다.

### 뭘 한 건가

**크로스 도메인 E2E 테스트 5개.** 지금까지는 "계약 API가 잘 되나?", "청구 API가 잘 되나?" 처럼 각각을 따로 테스트했다. Phase 4에서는 실제 업무 흐름 전체를 테스트했다:

1. 계약 등록 → 조회 → 목록에 포함되는지
2. 입주 → 퇴실 → 퇴실 기록 이동 → 공실 등록 (전체 생명주기)
3. 청구서 생성 → 확정 → 발송 → 현황 확인
4. 입금/지출 등록 → 거래 목록 확인
5. 연체 감지 → 실시간 알림 발송

### 백엔드가 완성됐다

Phase 3~4에서 만든 서버: API 48개, 테스트 105개 전부 통과, 테이블 20개. 프론트엔드는 한 줄도 안 건드렸다. 다음 Phase 5에서 프론트와 백엔드를 연결한다.

## 신규 파일 포팅

대표님 코드 통합 Phase 1을 비유하면 — **옆 건물에서 쓰던 가구와 설비를 우리 건물로 옮겨 들인 것이다.** 대표님이 별도로 만든 페이지, 컴포넌트, 데이터 저장소를 우리 구조에 맞게 변환해서 가져왔다.

### 뭘 한 건가

대표님은 우리가 재구축하는 동안 독자적으로 기능을 추가했다. Supabase(외부 DB)에 실 데이터를 넣고, 새 페이지 6개, 컴포넌트 7개, 데이터 저장소 9개를 만들었다. 이 코드는 JavaScript + Supabase 직접 호출 방식이라 우리 TypeScript + Spring Boot API 구조와 호환되지 않았다.

405개 파일이 달라졌지만, 자동 merge가 불가능했다. "대표님이 뭘 추가했나"를 기능 단위로 분리해서, 우리 구조에 맞게 하나씩 옮겼다.

**새 페이지 6개** — 회사설정(CompanySettingsPage), 퇴실링크(MoveOutLinkPage), 중개(BrokerPage), 건물관리(BuildingsPage), 급여(PayrollPage), 직원관리(StaffPage). 전부 `.jsx`에서 `.tsx`로 전환했다.

**새 컴포넌트 7개** — 검색창(SearchInput), 토스트 알림(Toast), 로딩 스피너(LoadingSpinner), 모달(Modal), 에러 화면(PageErrorBoundary), 건물 폼(BuildingFormSections), 호실 폼(RoomFormSection). 대표님이 현장에서 필요해서 만든 UI 부품들이다.

**Zustand 스토어 9개** — 대표님은 Zustand라는 상태 관리 라이브러리로 데이터를 관리했다. 건물, 임차인, 캘린더, 재무, 대기열 등. 우리는 TanStack Query로 서버 데이터를 관리하는데, 대표님 코드가 이 스토어에 의존하기 때문에 빌드를 위해 가져왔다. 두 체계가 공존하는 상태.

**Supabase 인프라** — 대표님의 Supabase 연결 코드(supabase.ts), 데이터 CRUD(supabaseData.ts), 사진 업로드(photoStorage.ts), 쿼리 훅(useSupabaseQueries.ts). 환경변수가 없으면 자동으로 비활성화되도록 방어 처리했다.

### 알려진 문제

BuildingsWrapper가 무력화됐다. 대표님의 BuildingsPage가 스토어에서 직접 데이터를 읽어서, 우리 wrapper가 props를 전달하는 구조가 작동하지 않는다. `VITE_USE_API=true` 경로가 죽어있다. Phase 2 Part 4에서 복구 예정.

## CalendarPage 기능 반영

Phase 2 Part 1을 비유하면 — **기존 캘린더 벽에 대표님이 현장에서 쓰던 도구 6가지를 옮겨 달고, 데이터 배관까지 전부 연결한 것이다.** 벽(디렉토리 구조)은 그대로 유지하면서, 도구만 하나씩 볼트로 고정했다.

### 뭘 한 건가

대표님의 CalendarPage(2,668줄)와 우리 calendar/ 디렉토리(15개 파일)를 비교해서, 대표님이 추가/변경한 기능 15개를 식별했다. 이 중 우리 구조에 반영할 것과 반영하지 않을 것을 분류하고, 4번의 Round에 걸쳐 반영했다.

**calendarApi.ts (데이터 배관 분기 밸브)** — 대표님은 Supabase에 직접 데이터를 쓴다. 우리는 Spring Boot 서버를 거친다. 이 두 경로를 한 곳에서 분기하는 "밸브"를 만들었다. `persistInsert`, `persistUpdate`, `persistDelete` 3개 함수가 그 밸브다. 각 컴포넌트에서는 밸브만 열면 되고, 물이 어디로 가는지(Supabase vs API)는 feature flag가 결정한다.

**ExternalCheckModal (퇴실체크 도구)** — 직원이 퇴실 현장에 가서 하는 일을 4단계로 정리한 모달이다. 입주 때 찍은 사진과 퇴실 때 상태를 비교하고, 벽지 훼손 같은 공제 항목을 입력하고, 전기/가스 계량기를 검침하고, 수리가 필요한 곳을 체크한다. 이게 끝나면 "퇴실체크 완료" 표시가 붙고, 해당 이벤트는 실수로 삭제되지 않도록 삭제 가드가 작동한다.

**DirectInputModal (직접입력)** — 임차인이 퇴실 링크를 통해 정보를 안 넣었을 때, 직원이 직접 비밀번호, 환불 계좌 정보를 입력하는 모달이다. 예금주 이름이 계약서의 임차인 이름과 다르면 "예금주 불일치" 경고가 뜬다.

**단기/일반 퇴실 분기** — 퇴실 처리할 때 "이 호실이 단기 임대인가, 일반 임대인가"를 자동으로 판별해서 다른 체크리스트를 보여준다. 단기는 청소/사진만, 일반은 계량기 검침/보증금 정산까지. 대표님이 현장에서 "단기랑 일반이 절차가 다른데 같은 화면이라 헷갈린다"는 문제를 해결한 기능이다.

**waterFee 통일** — 대표님이 `water`를 `waterFee`로 바꿨다. 13개 파일에서 필드명을 맞췄다. 단기 임대 호실의 수도 요금 관련 필드인데, SMS 템플릿부터 공실 정보, 계약서 입력까지 전부 하나로 통일됐다.

**Supabase 연동 13개 파일** — 캘린더에서 이벤트를 추가/수정/삭제/드래그하는 모든 곳에 Supabase 배관을 연결했다. 이벤트를 만들면 Supabase에도 저장되고, 드래그해서 날짜를 옮기면 Supabase에도 반영된다. 사진을 올리면 Supabase Storage에도 올라간다. 전부 feature flag로 분기돼 있어서, 우리 API 서버를 켜면 Supabase 배관은 자동으로 닫힌다.

**건물목록 소스 보강** — 캘린더 필터의 건물 드롭다운에 "주소록"을 하나 더 연결했다. 기존에는 고정 목록 + 임차인/공실이 있는 건물만 나왔는데, 이제 시스템에 등록된 모든 건물이 나온다. 새 건물을 추가했는데 아직 임차인이 없어도 캘린더에서 선택할 수 있다.

### 뭘 안 건드렸나

대표님이 props를 제거하고 Zustand 스토어에서 직접 데이터를 읽도록 바꾼 건 반영하지 않았다. 우리는 wrapper가 데이터를 가져와서 props로 전달하는 패턴을 유지한다. 이 패턴이 있어야 feature flag로 "API 서버에서 읽기 vs localStorage에서 읽기"를 전환할 수 있기 때문이다.

calendar/ 디렉토리 분해 구조(15개 파일)도 그대로 유지했다. 대표님의 단일 CalendarPage.jsx(2,668줄)로 되돌리지 않았다.

## TenantsPage 기능 반영

Phase 2 Part 2를 비유하면 — **임차인 관리 데스크에 필터 서랍장을 추가하고, 대표님이 현장에서 쓰던 체크리스트와 자동 연결한 것이다.** 데스크(디렉토리 구조)는 그대로 두고, 서랍(필터)과 연결 파이프(데이터 연동)를 달았다.

### 뭘 한 건가

대표님의 TenantsPage(2,939줄)와 우리 tenants/ 디렉토리(11개 파일)를 비교해서, 반영할 기능을 3단계(로직 → Supabase → UI)에 걸쳐 적용했다.

**필터 보강** — 기존에는 이름/건물/호실 텍스트 검색만 가능했다. 이제 건물별 드롭다운(아페이론, 스타빌 등)과 상태 토글(전체/정상/연체)이 추가됐다. "스타빌 건물의 연체 임차인만 보기"가 한 번에 된다. 초기화 버튼도 있어서 필터를 한 번에 해제할 수 있다.

**퇴실체크 데이터 자동 연동** — CalendarPage에서 만든 ExternalCheckModal(퇴실체크)과 DirectInputModal(직접입력)의 데이터가 이제 TenantsPage로 자동으로 흘러온다. 직원이 퇴실 현장에서 공제 항목(벽지 훼손 3만원, 청소비 5만원)을 입력하면, 임차인 퇴실 정산 화면에 그 금액이 자동으로 채워진다. 환불 계좌 정보도 마찬가지. 기존에는 퇴실 문자 메시지를 파싱해서 추출했는데, 이제 직접 입력한 데이터가 우선 적용된다.

**tenantsApi.ts (데이터 배관 밸브)** — CalendarPage의 calendarApi.ts와 동일한 패턴. 재계약, 퇴실, 상세 수정, 신규 입주 — 4가지 동작에 Supabase 배관을 연결했다. feature flag로 분기돼 있어서 우리 API 서버를 켜면 자동으로 닫힌다.

**DB 소스 배지** — Supabase에서 가져온 임차인 데이터에는 초록색 "DB" 배지가 붙는다. 리스트에서도, 상세 화면에서도 보인다. 대표님이 Supabase에 직접 넣은 실 데이터와 localStorage 데이터를 시각적으로 구분할 수 있다.

**임차인연결 (매물등록)** — 일반임대/근생 호실에만 나타나는 기능. "이 호실이 매물로 나와있다"는 표시와 입주 가능일을 등록할 수 있다. 잠금/해제 토글이 있어서 실수로 변경하는 걸 방지한다. 부동산 중개인한테 "이 호실 언제부터 입주 가능합니다"라고 안내할 때 쓴다.

**다중 차량 (근생 최대 5대)** — 근생(상업용) 호실은 차량이 여러 대일 수 있다. 기존에는 1대만 등록 가능했는데, 이제 최대 5대까지 등록할 수 있다. 주차 유형(선착순/지정무료/리모컨/주차비 등)별로 주차비와 리모컨 보증금을 따로 관리하고, 총 비용을 요약 배너로 보여준다.

### 뭘 안 건드렸나

tenants/ 디렉토리 분해 구조(11개 파일)를 그대로 유지했다. 대표님의 단일 TenantsPage.jsx(2,939줄)로 되돌리지 않았다. 클라우드 사진/계약서 업로드(Round 4)는 별도 PR로 분리했다.

## AppLayout + 라우터 등록

Phase 2 Part 3을 비유하면 — **건물 안내판에 새 사무실 번호를 추가하고, 외부 방문자용 별도 출입구를 만든 것이다.** 건물 구조(AppLayout)는 그대로 두고, 대표님이 새로 만든 방(페이지)에 번호를 붙여서 찾아갈 수 있게 했다.

### 뭘 한 건가

Phase 1에서 가져온 신규 페이지 6개 중 4개(StaffPage, BrokerPage, PayrollPage, BuildingsPage)는 이미 라우터에 등록돼있었다. 나머지 2개를 처리했다.

**회사설정 페이지 (CompanySettingsPage)** — 사이드바 "설정" 섹션에 "하우스맨 기본정보" 메뉴를 추가했다. 직원이 클릭하면 `/company-settings` 경로로 이동한다. CompanySettingsWrapper를 만들어서 우리 wrapper 패턴에 맞췄다. 이 페이지는 내부 직원만 접근 가능하다 (인증 필요).

**퇴실링크 페이지 (MoveOutLinkPage)** — 이건 다른 페이지와 성격이 다르다. 임차인이 직접 접근하는 외부 페이지다. 퇴실할 때 직원이 임차인에게 SMS로 링크를 보내는데, 임차인이 그 링크를 열면 비밀번호, 환불 계좌 같은 정보를 직접 입력할 수 있다. 임차인은 로그인이 없으니까, 이 페이지는 AppLayout 바깥에 공개 라우트(`/move-out/:eventId`)로 등록했다.

**pageIdToPath 매핑** — 대표님 코드에서 `setPage("tenants")` 같은 방식으로 페이지를 전환하는 곳이 있다. 우리는 React Router를 쓰니까 `navigate("/tenants")`로 변환해야 한다. 이 변환 테이블(pageId → URL 경로)을 navigation.ts에 공식 export로 추가했다. 모든 페이지 ID가 정확한 URL로 매핑된다.

### 뭘 안 건드렸나

**AiChatBot** — 대표님이 AppLayout에 사이드 패널로 넣어둔 AI 챗봇이 있다. 하지만 이 컴포넌트가 Supabase 스토어에 직접 의존하고 있어서, 지금은 포팅하지 않았다. 별도 파트로 처리 예정.

AppLayout 자체의 구조(사이드바, 상단 바, 모바일 탭)는 변경하지 않았다. 라우터 경로와 네비게이션 메뉴만 추가했다.

## 코드 전수 분석 + 허점 보완

전수 분석을 비유하면 — **이사 후 짐을 다 풀었는데, 빠진 상자가 없는지 창고를 다시 한번 뒤진 것이다.** 405개 파일을 하나씩 대조해서 아직 안 옮긴 짐(미반영 기능)이 있는지 확인했다.

### 뭘 한 건가

**파일 단위 전수 점검** — 대표님이 수정한 파일 13개, 추가한 파일 96개를 전부 대조했다. 수정 파일 중 11개는 이미 반영 완료. 추가 파일 중 94개는 이미 포팅 완료. 빠진 건 2개: BuildingDetailPage(건물 상세 — 대표님이 2,292줄 분량으로 대폭 확장)와 AiChatBot(AI 챗봇 사이드패널).

**줄 단위 허점 점검** — CalendarPage와 TenantsPage는 "파일은 처리했으니 OK"가 아니라, 함수 이름 하나하나, 비즈니스 키워드(depositConfirmed, externalCheckDone, refundBank 등) 하나하나를 대조했다. 결과: 둘 다 CLEAN. 누락된 함수 없음, 누락된 키워드 없음.

**런타임 위험 3건 발견 + 즉시 수정:**

**Supabase null 가드** — 대표님 코드의 Supabase 함수 22개가 전부 "Supabase 서버가 연결 안 돼 있으면?" 상황을 안 다루고 있었다. 우리는 Supabase 없이도 돌아가야 하니까, 모든 함수 첫 줄에 "연결 안 돼 있으면 빈 값 돌려줘"라는 안전망을 추가했다. 수도꼭지를 열었는데 물이 안 나오면, 터지는 게 아니라 "물 없음"이라고 알려주는 것이다.

**BuildingsWrapper 복구** — Phase 1에서 대표님의 BuildingsPage를 가져올 때, 이 페이지가 Zustand 스토어에서 직접 데이터를 읽는 구조라 wrapper를 무력화시켰었다. 이번에 BuildingsPage를 props 방식으로 전환하고, BuildingsWrapper를 CalendarWrapper/TenantsWrapper와 동일한 패턴(useAppContext + useApiOr)으로 복원했다. 이제 VITE_USE_API=true(API 서버 모드)에서도 빈 화면이 아니라 정상 렌더링된다.

**BrokerWrapper 복구** — BrokerPage도 동일한 문제. `useCalendarStore.getState()`로 캘린더 이벤트를 직접 읽고 있었는데, 스토어에 데이터가 안 들어가 있으면 빈 배열만 나온다. calendarEvts를 props로 전환하고, BrokerWrapper에서 실제 데이터를 전달하도록 수정했다.

### 아직 남은 것

BuildingDetailPage — 대표님이 건물 상세 페이지에 편집 모드, 호실 추가/삭제, 건물 폼 6개 섹션, Supabase 연동 23건을 추가했다. 우리 buildings/detail/ 디렉토리(8개 파일, 358줄)에 아직 반영 안 됐다. 별도 Part로 처리 예정.

AiChatBot — 대표님이 AppLayout에 넣어둔 AI 챗봇 사이드패널. Supabase 스토어에 직접 의존해서 격리 작업이 필요하다. 핵심 워크플로우가 아니라 우선순위 낮음.

## BuildingDetailPage Supabase 통합

Phase 2 Part 5를 비유하면 — **건물 관리 사무실에 있는 서류함에 네트워크 케이블을 연결한 것이다.** 서류함(건물 상세 페이지) 자체는 이미 완성돼있었고, 서류(데이터)도 다 들어있었다. 다만 "서류를 넣거나 꺼낼 때 본사 DB에도 동기화되는" 배관이 없었다. 그걸 연결했다.

### 뭘 한 건가

전수 분석에서 BuildingDetailPage가 최대 미반영 항목으로 나왔었다. 대표님이 1,469줄로 확장한 반면 우리는 8개 파일 2,642줄. 분석해보니 기능 95%는 이미 있었고, 빠진 건 Supabase 배관 5개뿐이었다.

**buildingDetailApi.ts (데이터 배관 밸브)** — CalendarPage의 calendarApi.ts, TenantsPage의 tenantsApi.ts에 이어 세 번째 밸브. 건물 정보 수정, 건물 삭제, 담당자 조회/배정, 호실 수정 — 5가지 동작에 Supabase 배관을 연결했다.

**건물 정보 수정 동기화** — 직원이 건물 주소, 건물주 정보, 계좌 설정 등을 수정하면 로컬에 즉시 반영되면서 동시에 Supabase에도 비동기로 저장된다. 저장 실패해도 로컬은 이미 반영돼있어서 사용자 경험에 영향 없다.

**건물 삭제 Supabase 연동** — 3단계 확인(경고 → 재확인 → 건물명 타이핑) 모달은 이미 있었다. 여기에 실제 Supabase deleteBuilding 호출을 추가했다. FK 캐스케이드로 호실, 임차인, AS 기록까지 함께 삭제된다.

**담당자 배정 Supabase 연동** — 건물 상세 페이지를 열면 Supabase에서 해당 건물의 담당자 목록을 가져와서 화면에 채운다. 직원이 담당자를 변경하면 Supabase에 즉시 UPSERT된다. 같은 역할에는 1명만 배정 가능.

**호실 수정 Supabase 연동** — 호실의 기준 보증금/월세/관리비 등을 수정하고 "저장"을 누르면 Supabase rooms 테이블에도 반영된다.

**편집 중 이탈 방지** — 편집 모드에서 브라우저 탭을 닫거나 다른 페이지로 이동하려 하면 "저장하지 않은 변경사항이 있습니다" 확인 팝업이 뜬다.
