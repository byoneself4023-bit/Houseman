# Houseman 재구축

## PWA 방향성 (앱처럼 쓰기)

Flutter나 React Native 같은 네이티브 앱은 이 프로젝트에서 불필요하다. 사용자가 8명 내부 직원이고, 앱스토어에 배포할 서비스가 아니기 때문이다.

지금 만들고 있는 건 **반응형 웹앱**이다. 모바일 브라우저에서 그대로 쓸 수 있고, 바텀탭이 있는 이유가 직원들이 현장에서 스마트폰으로 접속할 때를 위한 것이다.

"앱처럼" 쓰고 싶으면 Phase 6(DevOps)에서 **PWA(Progressive Web App)** 설정을 추가하면 된다. 필요한 건 딱 2개:

**manifest.json** — 앱 이름, 아이콘, 테마 색상, 시작 URL을 정의하는 JSON 파일. 이게 있으면 모바일 브라우저에서 "홈 화면에 추가" 했을 때 앱 아이콘이 생기고, 실행하면 주소창 없이 전체 화면으로 뜬다.

**Service Worker** — 백그라운드에서 돌아가는 스크립트. 네트워크 요청을 가로채서 캐시에서 응답할 수 있어서, 오프라인이거나 네트워크가 불안정해도 앱이 뜬다. 직원들이 건물 현장(지하, 엘리베이터 등)에서 접속할 때 유용하다.

Phase 6에서 `vite-plugin-pwa` 하나 설치하면 둘 다 자동 생성된다. 코드 2~3줄 추가 수준. 8명 직원한테 "이 링크 열어서 홈 화면에 추가하세요" 한마디면 끝이다.

Flutter/React Native가 필요한 시점은 "카메라로 계량기 사진 찍어서 OCR로 자동 입력", "푸시 알림", "오프라인 모드" 같은 네이티브 기능이 필수일 때인데, 지금 Houseman 요구사항에는 없다.

## FE 툴링 기반

Phase 0에서 한 걸 비유하면 — **집은 그대로 두고 배관, 전기, 기초 공사를 한 것이다.**

### 뭘 한 건가

대표님이 만든 프로토타입은 JavaScript로 되어있었다. JavaScript는 "자유로운" 언어라 오타를 써도, 잘못된 데이터를 넣어도 실행하기 전까지 모른다. 실행해봐야 "터진다".

Phase 0에서 **TypeScript**를 씌웠다. 이건 "실행 전에 실수를 잡아주는 안전망"이다.

예를 들어:

```
// JavaScript — 실행해봐야 터지는 거 알 수 있음
tenant.bulding  // ← "building" 오타인데 아무도 안 잡아줌. 런타임에 undefined

// TypeScript — 코드 쓰는 순간 빨간줄
tenant.bulding  // ← 빨간줄: "bulding"은 Tenant에 없습니다. "building" 아닌가요?
```

42개 건물, 500명 임차인, 정산 설정 같은 **모든 데이터 구조를 타입으로 정의**했다 (434줄). 이제 이 데이터를 다루는 코드에서 오타나 잘못된 접근을 하면 빌드 자체가 안 된다. 실행해보기 전에 잡아준다.

### 같이 설치한 도구들

**ESLint** — "이 코드 이상한데?" 알려주는 도구. 안 쓰는 변수, React 훅 잘못 쓴 것, 위험한 패턴 등을 자동 감지한다.

**Prettier** — 코드 스타일 자동 통일. 들여쓰기, 따옴표, 세미콜론 같은 걸 한 번에 맞춰준다. 여러 명이 코드를 만져도 스타일이 동일하게 유지된다.

**Vitest** — 자동 테스트. "초성 검색이 잘 되나?", "금액 포맷이 맞나?" 같은 걸 코드가 자동으로 확인해준다. 14개 테스트가 돌아가고, 코드를 바꿀 때마다 "기존 기능이 안 깨졌는지" 자동 확인한다.

### 뭘 안 건드렸나

대표님이 만든 25개 페이지(임차인 관리, 건물 상세, 캘린더 등)와 App.jsx는 **한 줄도 안 건드렸다.** 데이터 값도 안 바꿨다. "안전망을 까는 작업"이지 "기능을 바꾸는 작업"이 아니다.

## FE 아키텍처 스켈레톤

Phase 1을 비유하면 — **방 하나짜리 원룸을 거실/침실/주방/욕실로 나눈 것이다.** 집 크기는 같은데 구조가 생겼다.

### 뭘 한 건가

대표님이 만든 App.jsx는 780줄짜리 단일 파일이었다. 로그인, 사이드바, 상단 바, 모바일 탭, 페이지 전환, 데이터 관리가 전부 한 파일에 들어있었다. 이걸 6개 파일로 쪼갰다:

- **AuthGate.tsx** — 로그인 화면
- **Sidebar.tsx** — 왼쪽 사이드바 메뉴
- **TopBar.tsx** — 상단 바 (사용자 정보, 시간)
- **MobileNav.tsx** — 모바일 하단 탭
- **AppLayout.tsx** — 전체 레이아웃 조합
- **router.tsx** — 페이지 경로 정의

### 사용자가 체감하는 차이 4가지

**1. URL이 생겼다**

이전에는 어떤 페이지를 가든 주소창이 `localhost:5173`으로 고정이었다. 지금은 사이드바에서 메뉴를 클릭하면 `/calendar`, `/tenants`, `/buildings` 이런 식으로 주소가 바뀐다.

**2. 뒤로가기가 된다**

이전에는 브라우저 뒤로가기를 누르면 아예 다른 사이트로 나갔다. 지금은 임차인정보 → 건물·호실정보로 이동한 뒤 뒤로가기를 누르면 임차인정보로 돌아온다.

**3. URL 직접 입력이 된다**

이전에는 `localhost:5173` 외에 아무 주소도 먹히지 않았다. 지금은 주소창에 `localhost:5173/tenants`를 직접 치면 바로 임차인 페이지로 간다. 이 URL을 다른 사람한테 공유할 수도 있다.

**4. 새 탭에서 열기가 된다**

이전에는 불가능했다. 지금은 `localhost:5173/buildings`를 새 탭에 붙여넣으면 건물 페이지가 바로 뜬다.

### 뭘 안 건드렸나

25개 페이지의 내부 코드는 한 줄도 안 건드렸다. 데이터 흐름(localStorage → 화면 표시)도 그대로다. "방을 나눈 것"이지 "가구를 바꾼 것"이 아니다.

### 레이아웃 세련화 

pages를 건드리지 않고 레이아웃 3파일(로그인, 사이드바, 상단 바)만 수정해서 첫인상을 바꿨다.

**로그인 화면** — 기존 1단 카드에서 2단 레이아웃으로 바뀌었다. 좌측에 브랜드 영역(HOUSEMAN 로고 + "건물관리의 새로운 기준" 슬로건), 우측에 로그인 폼. 모바일에서는 자동으로 1단으로 전환된다.

**사이드바** — 현재 보고 있는 메뉴 옆에 파란색 세로 바가 표시된다. 이전에는 작은 파란 점이었는데 더 눈에 잘 띈다. 메뉴 간격이 약간 넓어져서 답답한 느낌이 줄었다.

**상단 바** — "박종호 대표" 텍스트 옆에 이니셜이 든 원형 아바타가 추가됐다. 우측에 있던 "🏗️ 관리" 표시는 삭제됐다 (사이드바 역할 탭으로 충분하므로). 알림 벨 아이콘 위에 마우스를 올리면 살짝 흔들린다.

## 백엔드 기본 인프라

Phase 3a를 비유하면 — **건물에 수도관과 전기선을 연결한 것이다.** 건물(프론트엔드)은 그대로인데, 뒤에서 물(데이터)이 흐를 수 있는 배관(서버)이 깔렸다.

### 뭘 한 건가

지금까지 모든 데이터는 브라우저의 localStorage에 저장됐다. 이건 "내 컴퓨터에만 있는 메모장"이다. 다른 직원이 볼 수 없고, 브라우저 캐시를 지우면 데이터가 사라진다.

Phase 3a에서 **Spring Boot 서버**를 만들었다. 이 서버가 **PostgreSQL 데이터베이스**에 데이터를 저장한다. 직원 9명, 건물 21개, 호실 200개를 데이터베이스에 넣었다. 값은 프로토타입과 정확히 동일하다.

### 서버가 하는 일

**로그인** — 전화번호와 비밀번호를 서버에 보내면, 서버가 확인하고 "출입증"(JWT 토큰)을 발급한다. 이 출입증이 있어야 데이터에 접근할 수 있다. 프로토타입에서는 비밀번호가 코드에 그대로 노출돼 있었는데, 이제 암호화(BCrypt)된 상태로 데이터베이스에 저장된다.

**건물/호실 조회** — 출입증을 제시하면 자기가 담당하는 건물 목록을 받을 수 있다. 총괄 직원은 전체 21개, 담당자는 자기 건물만 보인다.

### 아직 프론트에는 연결 안 됐다

서버는 만들어졌지만, 프론트엔드는 아직 localStorage를 보고 있다. Phase 5에서 스위치를 켜면(VITE_USE_API=true) 프론트가 서버에서 데이터를 가져오게 된다.

## 핵심 비즈니스

Phase 3b를 비유하면 — **건물 배관에 실제 물을 흘려보낸 것이다.** Phase 3a에서 배관(서버)을 깔았고, 이제 계약/거래/청구라는 실제 업무 데이터가 흐르기 시작했다.

### 뭘 한 건가

프로토타입에서 가장 중요한 3가지 데이터를 서버로 옮겼다.

**계약** — 임차인 488명의 계약 정보(이름, 전화번호, 보증금, 월세, 관리비, 입주일, 만료일, 연체 상태). 퇴실 기록 14건도 포함.

**거래** — 입금/지출 내역 7건. 어떤 건물, 어떤 호실에서 얼마가 들어오고 나갔는지.

**청구** — 이 부분이 가장 복잡하다. 42개 건물마다 정산 방식이 다르다(수수료율 5%, 월급형 150만원, 고정형 50만원 등). 호실마다 전기/가스/수도 설정이 다르다(계량기 번호, 단가, 검침 기간 등). 이 설정을 전부 데이터베이스에 넣었다.

### 정산 계산 로직 보존

대표님이 만든 billingMaster.ts의 계산 함수 5개를 Kotlin으로 그대로 옮겼다. 10원 단위 절사, 연체수수료(납기 5일 초과 시 5%), 수수료 계산, 일할 계산, 부가세 계산. 한전 누진 3구간 요금 계산도 포함. **같은 숫자를 넣으면 같은 결과가 나온다.**

### 청구서 생성이 서버에서 된다

이전에는 브라우저에서 계산했다. 이제는 서버에 "제이앤제이 2026년 3월 청구서 만들어줘"라고 요청하면, 서버가 해당 건물의 계약 정보 + 공과금 설정을 조합해서 호실별 청구서를 자동 생성한다. 생성(DRAFT) → 확정(CONFIRMED) → 발송(SENT) 단계로 관리.

## 파생 도메인 + 실시간 알림

Phase 3c를 비유하면 — **건물에 인터폰과 CCTV를 설치한 것이다.** 배관(서버)으로 물(데이터)이 흐르고 있는 상태에서, 이제 "누가 입주했다", "연체가 발생했다" 같은 알림이 실시간으로 울리게 됐다.

### 뭘 한 건가

**캘린더** — 입퇴실 일정과 직원 휴무를 관리한다. "3월 1일 아페이론 101호 퇴실", "3월 7일 유은혜 부장 휴무" 같은 이벤트를 등록/수정/삭제할 수 있다.

**공실 관리** — 비어있는 호실을 등록하고 상태(점검/청소중, 홍보중 등)를 관리한다. 보증금, 월세, 중개 수수료 정보도 포함.

**건물주 정산** — 이 부분이 가장 복잡하다. 건물마다 정산 방식이 다르다. 수수료형(5%), 월급형(150만원), 수금형 등. 서버에 "스타빌 2026년 3월 정산해줘"라고 요청하면, 해당 건물의 16개 호실 임대료에서 수수료를 계산하고, 이번 달 입퇴실 정산, 공과금 공제까지 반영한 최종 정산서를 만들어준다.

**실시간 알림(SSE)** — 직원이 로그인하면 서버와 연결이 유지된다. 다른 직원이 계약을 등록하면 "스타빌 105호 입주 예정: 홍길동" 알림이 즉시 뜬다. 청구서가 확정되면, 입금이 들어오면, 연체가 감지되면 — 모두 실시간 알림.

**자동 감시** — 매일 아침 9시에 서버가 연체 계약을 자동 감지해서 알림을 보낸다. 매월 1일에는 30일 이내 만료되는 계약을 알려준다. 직원이 따로 확인할 필요 없이 서버가 알아서 챙겨준다.

## 부가 기능

Phase 3d를 비유하면 — **건물에 우편함과 주차장 관리대를 놓은 것이다.** 핵심 배관(정산, 청구, 계약)은 이미 완성됐고, 부가 시설을 추가한 것이다.

### 뭘 한 건가

**출납장(Cashbook)** — 건물별 입금/출금을 기록하는 장부. 정산 후 건물주에게 송금할 때, 수리비를 지출할 때 등을 기록한다. "대기 → 완료 → 보류" 상태 관리가 있어서 송금 전/후를 추적할 수 있다.

**주차 관리** — 임차인별 차량 정보(차량번호, 차종)를 관리한다. 어떤 건물, 어떤 호실의 임차인이 어떤 차를 가지고 있는지 기록.

**엑셀 업로드는 스킵** — 대표님이 만든 엑셀 업로드 기능은 브라우저에서 직접 localStorage에 쓰는 방식이라 서버 API가 필요 없다. Phase 5에서 서버 연결할 때 같이 처리.

### Phase 3 전체가 끝났다

Phase 3a(기본 인프라) → 3b(계약/거래/청구) → 3c(캘린더/공실/정산/실시간 알림) → 3d(출납/주차)까지 완료. 서버에 53개 API가 만들어졌고, 98개 테스트가 전부 통과한다. 프론트엔드는 한 줄도 안 건드렸다.

## Backend API 마무리

Phase 4를 비유하면 — **건물 준공 검사를 한 것이다.** 배관(서버), 전기(인증), 인터폰(실시간 알림), 우편함(출납), 주차장(주차 관리)까지 다 설치한 뒤, "입주부터 퇴실까지 전체 흐름이 정상 작동하는가"를 테스트했다.

### 뭘 한 건가

**크로스 도메인 E2E 테스트 5개.** 지금까지는 "계약 API가 잘 되나?", "청구 API가 잘 되나?" 처럼 각각을 따로 테스트했다. Phase 4에서는 실제 업무 흐름 전체를 테스트했다:

1. 계약 등록 → 조회 → 목록에 포함되는지
2. 입주 → 퇴실 → 퇴실 기록 이동 → 공실 등록 (전체 생명주기)
3. 청구서 생성 → 확정 → 발송 → 현황 확인
4. 입금/지출 등록 → 거래 목록 확인
5. 연체 감지 → 실시간 알림 발송

### 백엔드가 완성됐다

Phase 3~4에서 만든 서버: API 48개, 테스트 105개 전부 통과, 테이블 20개. 프론트엔드는 한 줄도 안 건드렸다. 다음 Phase 5에서 프론트와 백엔드를 연결한다.
