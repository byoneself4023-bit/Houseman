## §[1] Phase 7 — n8n 자동화

**영역**: 운영 알림 / 정기 작업 자동화.

| 자동화 | 트리거 | 액션 |
|---|---|---|
| 임차료 알림 | 매월 1일 09:00 | 청구 대상 임차인에게 카카오톡 메시지 자동 발송 |
| 계약 만료 알림 | 만기 7일 전 09:00 | 운영자 + 임차인에게 알림 |
| 월정산 자동 | 매월 1일 00:00 | 전월 정산 자동 계산 + 건물주에게 PDF 첨부 메일 |
| 연체 독촉 | 단기 연체 5일 이상 매일 | 임차인 카카오톡 + 운영자 알림 |
| 공실 30일 이상 | 매주 월요일 | 운영자에게 홍보 전략 검토 알림 |

**산출물**:
- n8n 워크플로우 5개 (위 5건)
- BE에서 n8n webhook 호출 또는 n8n이 BE API polling
- 카카오톡 비즈니스 API 연동 (별도 비즈니스 채널 인증)

**진입 조건**: Phase 6 완료 + 카카오톡 비즈니스 채널 인증 보유.

**산출물 목표**: 운영자 일일 알림 작업 80% 자동화 (수동 알림 20% — 이슈 대응만).

## §[2] Phase 8 — 이카운트 OAPI (회계 통합)

**영역**: 입금/지출 데이터 이카운트 회계 시스템 자동 동기화.

**현재**: 운영자가 매일 BE 입출금 데이터 → 이카운트 엑셀 수동 입력 (1~2시간/일).

**목표**: BE `Transaction` 변경 시 이카운트 OAPI 자동 호출.

**산출물**:
- 이카운트 OAPI 연동 (인증 토큰 / 거래 등록 엔드포인트)
- BE Transaction 후처리 hook (저장 후 이카운트 자동 동기화)
- 동기화 실패 시 운영자 알림 + 재시도 큐
- 양방향 동기화 (이카운트에서 추가된 거래 → BE 반영)

**진입 조건**: 이카운트 OAPI 계약 + 회계사와 정합성 검토.

**산출물 목표**: 운영자 회계 입력 작업 95% 자동화.

## §[3] Phase 9 — OCR / Vision Model (영수증 / 한전 PDF)

**영역**: 비정형 문서 자동 데이터 추출.

**현재**: 운영자가 한전 청구서 PDF → 사용량 / 요금 수동 입력. 영수증 사진 → 출납장 수동 입력.

**목표**: PDF/이미지 업로드 → 자동 추출 → BE 자동 등록.

**산출물**:
- OCR API 연동 (네이버 클로바 OCR / Google Vision)
- 한전 PDF 전용 파서 (사용량 / 요금 / 검침일 추출)
- 영수증 이미지 파서 (금액 / 일자 / 항목 추출)
- 추출 결과 확인 모달 (운영자 1회 검수 후 저장 — 정확도 보장)

**진입 조건**: Phase 8 완료 + 한전 PDF 30 sample 수집.

**산출물 목표**: 한전 입력 작업 100% 자동화 / 영수증 입력 80% 자동화.

## §[4] Phase 10 — Qdrant RAG (계약서 / 매뉴얼 조회)

**영역**: 비정형 텍스트 (계약서 / 사내 매뉴얼 / 과거 운영 기록) 자연어 조회.

**현재**: 운영자가 특수 케이스 (예: 계약서 조항 / 과거 운영 사례) 발생 시 파일 폴더 수동 검색.

**목표**: 자연어 질의 → 관련 문서 + 답변 자동 제공 (RAG).

**산출물**:
- Qdrant 벡터 DB 구축
- 임베딩 파이프라인 (계약서 PDF / docs/ / 과거 운영 메모)
- 운영자용 채팅 UI (사이드바)
- 답변 source 명시 (인용 문서 + 페이지)

**진입 조건**: 계약서 100건 이상 + 사내 매뉴얼 정리 완료.

**산출물 목표**: 특수 케이스 답변 시간 평균 30분 → 30초.

## §[5] Phase 11 — AGENTS.md + ADR 정착 + 무인 운영 영역

**영역**: Agent Harness 시스템 자체 진화 + 영구 룰 자동 도출.

**현재**: AGENTS.md / CLAUDE.md / docs/ 매뉴얼이 운영자 수동 갱신. Architecture Decision Record (ADR) 미도입.

**목표**: 카드 진행 결과가 자동으로 ADR 누적 + AGENTS.md 자동 갱신.

**산출물**:
- ADR 템플릿 + 자동 생성 스크립트 (카드 commit hash + 결정 옵션 + 트레이드오프 자동 추출)
- AGENTS.md 수치 영역 (BE API / @Test 등) 자동 갱신 Hook
- 무인 운영 영역 정의 — Phase 1~10 진행 중 운영자 의사결정 0건 카드 영역 식별
- "Subagent → Subagent 자동 호출" 1단계 nesting 완화 시도 (Anthropic API 갱신 시)

**진입 조건**: Phase 7~10 진행 후 카드 30건 이상 누적 (패턴 분석 가능).

**산출물 목표**: 운영자 카드당 액션 횟수 3~4회 → 1~2회.

## §[6] Phase 12 — 운영 데이터 시각 검토 (Grafana / 자체 대시보드)

**영역**: 운영 KPI 실시간 시각화 + 이상치 자동 알림.

**현재**: 운영자가 매월 정산 후 엑셀로 KPI 수동 정리.

**목표**: BE 데이터 → Grafana 또는 자체 대시보드 실시간 표시. 이상치 자동 알림.

**산출물**:
- Grafana + Prometheus 연동 (또는 자체 React 대시보드)
- KPI 표시: 점유율 / 평균 임대료 / 연체율 / 정산 D 타입 수금률 / 공실 평균 일수
- 이상치 알림: 점유율 5% 이상 감소 / 연체율 10% 이상 증가 / 수금률 80% 미만
- 건물주별 대시보드 (Owner.owned_by 관계 활용, 본인 건물만 표시)

**진입 조건**: Phase 11 진행 + 운영 데이터 1년 이상 누적.

**산출물 목표**: 월별 KPI 정리 작업 8시간 → 0시간 (실시간 자동).

## §[7] 진화 순서 의존성

```
Phase 1~6 (CLAUDE.md + Apply.md §[4])
   ↓
Phase 7 (n8n) — 운영자 알림 자동화 우선
   ↓
Phase 8 (OAPI) — 회계 통합 (Phase 7 알림 + Phase 8 입력 = 운영자 일일 작업 50% 감소)
   ↓
Phase 9 (OCR) — 비정형 문서 처리 (Phase 8 후 정형 데이터 처리 완성된 상태에서 진입)
   ↓
Phase 10 (RAG) — 비정형 조회 (Phase 9 후 데이터 누적 완성)
   ↓
Phase 11 (ADR + 무인) — 시스템 자체 진화 (Phase 7~10 패턴 분석 가능)
   ↓
Phase 12 (대시보드) — 운영 KPI 시각화 (모든 데이터 누적 완성 후)
```

병렬 가능: Phase 9 ↔ Phase 10 (OCR과 RAG는 독립 영역).
